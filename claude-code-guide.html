★代理循环:它怎么"转"

你给一个任务,Claude 不是"想一下直接吐结果",而是像一个真的在干活的人一样,不断"查资料 → 动手 → 检查"。

三个阶段会反复交织:一个问答可能只需要"收集上下文";一次修 bug 会把三步来回跑好几轮;一次重构可能光"验证"就要跑很多遍。
💡 您全程都在这个循环里:按一次 Esc 立即打断当前操作,工具调用会被取消,Claude 停下等你说话;直接打字发新指令并回车,不用等它说完 —— Claude 会在手头这个操作做完后立刻读到,并调整方向。
🎨 Figma 类比: 有点像你在改一版设计稿:先打开参考图和现有组件看清楚现状(收集上下文),再动手改(采取行动),改完对齐一下间距、检查一遍交互(验证结果)。你随时可以在旁边插一句"等等,先别改这块",对方会立刻停下来听你说。

驱动这个循环的是两样东西:模型负责"想"(推理该做什么),工具负责"做"(真的去改文件、跑命令)。Claude Code 本身是包在模型外面的一层代理框架 —— 它负责准备工具、管理上下文、提供一个能真正执行动作的环境,把一个"语言模型"变成一个"能写代码的代理"。

1模型与工具:谁在想,谁在做

模型负责推理,工具负责动手。每用一次工具,Claude 都会拿到新信息,再决定下一步 —— 这就是"代理循环"落到实处的样子。

模型:负责"想"

Claude 能读懂任何语言写的代码,理解模块之间怎么连接,并想清楚要改什么才能达到你的目标。复杂任务会被拆成一步步来,边做边根据学到的信息调整。日常任务用 Sonnet 就够快够好;涉及复杂架构决策时可以切到 Opus,推理更强。会话中用 /model 随时切换。

工具:负责"做"

没有工具,Claude 只能用文字回你话;有了工具,它才能真的读文件、改文件、跑命令、查网络、连外部服务。内置工具大致分五类:

类别Claude 能做什么
📄 文件操作读文件、改代码、新建文件、重命名和重新整理目录结构
🔎 搜索按文件名找、按内容用正则搜、探索整个代码库
⚙️ 执行跑 shell 命令、启动服务、跑测试、操作 git
🌐 网络搜网页、抓文档、查报错信息
🧠 代码智能改完看类型报错、跳转到定义、查引用(需要装代码智能插件)
💡 除了这些基础工具,Claude 还能用 skills 扩展知道的东西、用 MCP 连外部服务、用 hooks 自动化流程、用 subagents 把任务外包出去。这些都是叠加在"核心循环"之上的能力,不属于内置工具本身。

一个真实例子:"修复失败的测试"

当你说"修复失败的测试"时,Claude 大致会这样跑:

1
跑一遍测试套件,看哪里失败了
2
读报错输出
3
搜索相关的源文件
4
读这些文件,理解代码逻辑
5
改文件,修问题
6
再跑一次测试,确认真的修好了
🎨 Figma 类比: 工具就像 Figma 里的选区工具、文字工具、评论区 —— 每次用一次都会给你新反馈(选中了什么、字对不对齐、同事回复了什么),这些反馈决定你下一步点哪里。Claude 用工具的逻辑是一样的。

2Claude 能看到什么

在项目目录里跑 claude 时,它能访问这些东西 —— 这也是它跟"只看当前这一个文件"的内联代码助手最大的区别。

📁 你的项目

目录和子目录里的所有文件,以及你额外授权过的其他位置的文件。

💻 你的终端

你能跑的任何命令:构建工具、git、包管理器、系统工具、脚本。你能在命令行做的事,它都能做。

🌿 你的 git 状态

当前分支、还没提交的改动、最近的提交历史。

📝 CLAUDE.md

你写的一份 markdown 说明书,放项目约定、规范,每次会话都会自动读。

🧠 自动内存

Claude 干活时自动记下的项目模式和你的偏好,会写进 MEMORY.md,每次会话开头自动加载前 200 行。

🔌 你配置的扩展

连外部服务的 MCP、封装工作流的 skills、能分工的 subagents、能操作浏览器的 Claude in Chrome。

因为看得到整个项目,Claude 能跨文件干活。你说"修复登录报错的 bug",它会自己去搜相关文件、读懂上下文、跨多个文件做协调修改、跑测试验证,如果你要求还会直接帮你 commit。

🎨 Figma 类比: 不是只看你正打开的这一个 frame,而是能翻遍整个文件的所有页面、组件库、你写在描述里的设计规范,甚至知道这个文件目前"谁改了哪块还没提交评审"。

3执行环境与使用界面

循环、工具、能访问的东西在哪里用都一样,变的只是"代码在哪跑"和"你用什么界面跟它说话"。

环境代码在哪跑适合场景
本地你自己的电脑默认方式,能完整访问你的文件、工具和环境
云Anthropic 托管的虚拟机把任务卸下去跑、处理本地没有的仓库
远程控制还是你的电脑,只是从浏览器远程操作想用网页界面,但代码还是留在本地
🎨 Figma 类比: 本地 = 在自己电脑上开着 Figma 文件改;云 = 让同事在 Figma 云端帮你跑一个很耗时的批量操作,不占你电脑资源;远程控制 = 用手机浏览器远程操作你电脑上正开着的那份 Figma。

访问入口有很多种:终端、桌面应用、IDE 插件(VS Code / JetBrains)、网页版 claude.ai/code、远程控制、Slack、CI/CD 流水线。界面只是决定你怎么看到、怎么跟它对话,底层那套"代理循环"是同一套。

4会话与上下文窗口

你干活的时候,整个对话会被实时存到本地。这一节讲会话怎么续、怎么分叉,以及"记性"用完了会发生什么。

会话是独立的

每条消息、每次工具调用、每个结果都被写进 ~/.claude/projects/ 下的纯文本文件,这也是"撤销""恢复""分叉"会话能实现的原因。但每个新会话都是从零开始的新上下文 —— 不会自动带上之前会话的聊天记录。跨会话的"记性"靠自动内存,你自己要长期生效的规则要写进 CLAUDE.md。

恢复(resume / --continue)→ 沿用同一个会话 ID,新消息接着往后写
分叉(fork / --fork-session / /branch)→ 复制出一份历史,开一个新会话 ID,原会话不受影响
🎨 Figma 类比: 恢复会话像"继续编辑同一份 Figma 文件";分叉像"另存为一个副本接着改",原文件保持不动,你可以放心在副本里做实验性改动。
💡 会话跟"你当前所在的目录"绑定。切分支时,Claude 看到的文件会跟着变,但聊天记录不会丢 —— 它还记得之前跟你聊过什么。想同时跑多个并行会话,可以配合 git worktree,给每个分支一个独立目录。

上下文窗口:会越用越满

上下文窗口里装着:对话历史、文件内容、命令输出、CLAUDE.md、自动内存、已加载的 skills、系统说明。用着用着会被填满,Claude 会自动"压缩":先清掉比较旧的工具输出,如果还不够再总结更早的对话。

你的原始请求和关键代码片段会被保留;对话早期的详细说明有可能在压缩里丢掉 —— 所以长期要生效的规则,别指望"聊天记录里说过",写进 CLAUDE.md。
💡 想看当前占用情况用 /context;想指定压缩时重点保留什么,可以带方向跑 /compact focus on the API changes,或者直接在 CLAUDE.md 里写一节 "Compact Instructions"。

用 skills 和 subagents 少占空间

Skills 是按需加载的:会话开始时 Claude 只看到 skill 的一句话描述,真正用到时才把完整内容加载进来。Subagents 更彻底 —— 它们在完全独立的上下文里干活,不会占用你主对话的空间,做完只给你一份摘要。

🎨 Figma 类比: Skill 像是收在组件库里但没展开的组件说明,要用到才展开细节;Subagent 像是把一部分设计工作外包给另一个人,在他自己的文件里完成,做完只给你一份效果说明,不会把他的草稿也塞进你的文件里。

5安全机制:撤销与权限

两道保险:出错了能撤回(checkpoint),以及提前定好哪些操作要问你、哪些不用问(权限模式)。

Checkpoint:每次改文件都能撤销

Claude 动手改任何文件之前,都会先给当前内容拍一张"快照"。改坏了,按两下 Esc 就能回到改之前的状态,或者直接叫它撤销。

⚠️ Checkpoint 只管"文件改动",跟 git 无关,也不会话与话之间共享。真正有外部副作用的操作(改数据库、调 API、上线部署)是没法拍快照撤销的 —— 所以 Claude 跑这类命令前会先问你。

权限模式:哪些事要先问你

按 Shift+Tab 在四种模式间切换:

四种模式管的是"要不要先问你";Plan Mode 里权限提示照样生效,只是它压根不会去动你的源文件。
💡 像 git status、npm test 这类你完全信得过的命令,可以在 .claude/settings.json 里加进允许列表,以后就不用每次都问了。设置范围可以从整个组织一路细化到你个人。

💬怎样把话说好

这几条来自官方文档,是能实打实提高第一次成功率的说话方式。

1

这是一场对话,不用一次说对

不需要憋一个"完美 prompt"。先说你想要的,不对就接着说,Claude 会在这句话上继续调整,不会推倒重来。

第一轮
修复登录报错。
发现方向不对,第二轮直接纠正
这不太对,问题出在会话处理那块,你再看看。
2

预先说清楚,越具体返工越少

提到具体文件、说明约束条件、指出可以参考的现有写法,第一次就做对的概率会高很多。

太模糊
结账流程坏了,帮我修一下。
具体明确
持卡已过期的用户走结账流程会报错。请检查 src/payments/ 里的问题,重点看令牌刷新那部分。先写一个能复现问题的失败测试,再动手修。
3

给它一个能自己核对的标准

有测试用例、有预期效果的截图、有明确的输出定义,Claude 就能自己检查做得对不对,而不是全靠你事后挑错。

带验证标准的需求
实现 validateEmail 函数。测试用例:'user@example.com' → true,'invalid' → false,'user@.com' → false。写完跑一下测试。
💡 涉及视觉效果的需求,直接把设计稿截图贴给它,让它对着截图检查自己实现出来的效果对不对。
4

复杂问题,先探索再动手

按两次 Shift+Tab 进入 Plan Mode,让 Claude 先只看代码、不动文件,出一份计划给你看,你确认后再执行 —— 比直接让它冲进去改代码效果更好。

Plan Mode 先出方案
读一下 src/auth/,搞清楚我们现在是怎么处理会话的。然后给加 OAuth 支持 这件事出一份计划。
5

委派,而不是逐条下指令

把它当成一个靠谱的同事:给背景和方向,细节交给它自己判断,不用手把手指定"读哪个文件、跑哪条命令"。

委派式描述
持卡已过期的用户走结账流程会坏。相关代码在 src/payments/。你能查一下并修好吗?
💡 不知道怎么用某个功能?直接问它。 比如"我怎么配置 hooks?""CLAUDE.md 应该怎么写比较好?",Claude 会直接解释给你听。/init 帮你搭 CLAUDE.md,/agents 帮你配 subagent,/doctor 排查安装问题。
