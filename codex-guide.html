Codex 使用指南

Codex 是 OpenAI 的编程代理,能自己读代码、动手改、跑命令、再检查一遍。它和 Claude Code 长得很像,这一页帮你搞懂它的用法,以及几种形态(命令行/IDE/云端)分别在什么场景用。

★代理循环:它怎么"转"

你给一个任务,Codex 不是"想一下直接吐结果",而是在一个循环里不断:问模型该干嘛 → 真的动手(读文件/改文件/跑命令) → 拿到结果再接着判断,直到任务完成。

一个简单问答可能只需要转一圈;一次修 bug 会来回转好几轮;复杂重构可能光"检查结果"就要跑很多遍。
🎨 Figma 类比: 跟你改一版设计稿的过程一样:先看参考、想清楚要改哪(调用模型),再动手拉组件、调间距(执行操作),改完对齐检查一遍交互(检查结果)。不满意就再来一轮。Codex 只是把这个循环自动跑起来。

整个循环由两样东西驱动:模型负责"想"(推理该做什么),工具负责"做"(真的去读文件、改文件、跑命令、搜网页)。你随时可以插话调整方向,不用等它跑完一整轮。

💡 Codex 和 Claude Code 底层是同一套"代理循环"思路,所以你在这两个工具之间切换时,说话方式几乎通用。区别主要在命令、几种运行形态、以及项目说明文件叫什么名字(Codex 叫 AGENTS.md,Claude 叫 CLAUDE.md)。

1Codex 能做什么

它不只是"帮你补全代码",而是能完整接手一个开发任务 —— 从看懂现有代码,到动手改,再到自己验证。

✍️ 写代码

你描述想要的东西,它生成符合现有项目结构和风格的代码。

📖 读懂代码

读并解释复杂的、别人写的、或很老的遗留代码,告诉你它在干嘛。

🔍 审查代码

分析代码找出潜在 bug 和逻辑问题,可以在提交前先审一遍。

🐞 调试修复

追踪失败在哪、诊断根本原因、给出有针对性的修复方案。

🔧 自动化任务

接手重构、写测试、做迁移这类重复又繁琐的活。

🖼️ 看图 + 联网

能读你贴的截图/设计稿、生成和编辑图片、搜网页、接 MCP 外部工具。

🎨 Figma 类比: 与其说它是"自动画图的插件",不如说它是一个能看懂整个 Figma 文件、按你的意思接手一块设计工作、做完还会自己对一遍规范的助手 —— 你贴张参考截图给它,它能照着来。

2四种形态:分别在哪用

同一个 Codex,包了好几层"外壳"。底层能力一样,区别只在"你在哪跟它说话""代码在哪跑"。

四种形态背后是同一个 Codex。选哪个,看你习惯在哪工作、以及这个任务想让它在本地还是云端跑。
形态在哪用什么时候选它
CLI 命令行你的终端最灵活,能写进脚本自动化;重度用户的主力方式
IDE 扩展VS Code 等编辑器里边写代码边用,它会自动带上你正打开的文件和选中的代码
Cloud 云端chatgpt.com/codex 网页把耗时任务丢给它在云端后台跑,还能同时跑好几个
App + 集成桌面应用 / GitHub / Slack在 GitHub issue 或 PR 里 @codex 直接触发;团队场景
💡 不用纠结"该学哪个"。先从一种开始(设计师一般从 IDE 扩展或网页版最省心),同一份 AGENTS.md 和同样的说话方式在四种形态里都通用。

3本地线程 vs 云端线程

一次对话叫一个"线程"(thread)。线程分两种,区别就是"代码到底在哪台机器上跑"。

本地线程像"你在旁边看着它改";云端线程像"把任务派出去,它在别处做完再把成果交给你审"。
🎨 Figma 类比: 本地线程 = 你和它共享同一个屏幕、一起改当前这份文件;云端线程 = 你把一块设计需求派给远程同事,他在自己那边做完,再开一个"待评审"给你看,你确认了才合进主文件。
💡 云端任务最省心的入口是 GitHub:在 issue 或 PR 里评论 @codex 加上你的要求,它就会在云端接手,做完直接提一个 PR。你只需要像审设计稿一样审它的改动。
⚠️ 让它联网前想一下:云端环境是否要访问公网、能用哪些工具,这些是你在环境配置里自己决定的。不确定就先关掉联网,只让它在仓库范围内干活。

4审批模式与沙箱

两道保险:沙箱(sandbox)把它的动作圈在一个安全范围内;审批模式(approval)决定哪些动作要先问你、哪些可以自己做。

沙箱:它的动作被圈在安全范围里

Codex 执行操作时是在一个沙箱里跑的 —— 相当于给它一个"隔离的工作台",它改文件、跑命令都在这个范围内,不会随便碰到范围外的东西。真正有风险的动作(比如联网、改沙箱外的文件)会被拦下来问你。

审批模式:哪些事要先问你

在动手改文件、跑命令之前,Codex 可以按你设的"审批级别"来决定要不要先征求你同意。你可以按自己对自动化的接受程度来调:

刚上手、或改的是重要代码时,用"每步都问";熟了、或是跑一批放心的重复任务时,再放宽。
🎨 Figma 类比: 就像给一个新来的实习生设权限:一开始"改任何东西都先问我",信任建立后再逐步开放"这类小改你自己来就行,大改动再喊我"。
💡 用 /model 可以随时切换模型或调整"推理强度"(想得更快还是想得更深)。任务简单就快一点,任务烧脑就让它多想一会。

5AGENTS.md:项目说明书

AGENTS.md 是你写给 Codex 看的一份说明书。每次它在这个项目里干活都会先读,省得你每次都重复交代同样的规矩。

它就是一个放在项目根目录的普通 Markdown 文件,写这个项目"怎么装依赖、怎么跑测试、代码风格是什么、提交信息怎么写"等等。没有必填字段,想写什么标题都行。已经有超过 6 万个开源项目在用这个格式。

## Setup commands
- 安装依赖: `pnpm install`
- 启动开发服务器: `pnpm dev`
- 跑测试: `pnpm test`

## Code style
- TypeScript strict mode
- 用单引号, 不加分号
🎨 Figma 类比: 就像你在团队文件最前面钉一页"设计规范说明":配色、字号、组件命名规则、间距系统。新来的人(或 AI)先看这一页,就不会乱来。AGENTS.md 就是给 Codex 看的那一页。
💡 可以分层放。 大项目里每个子文件夹都能有自己的 AGENTS.md,Codex 会自动读离它正在改的文件最近的那一份。所以子模块可以有自己更具体的规矩,不冲突。
💡 优先级: 离改动文件最近的 AGENTS.md 优先;而你当场在对话里说的话,永远盖过 AGENTS.md 里写的。所以临时想破个例,直接说就行。
让 Codex 帮你起草 AGENTS.md
帮我在项目根目录创建一份 AGENTS.md。先看一下项目结构和 package.json,总结出:怎么装依赖、怎么启动、怎么跑测试,以及你观察到的代码风格约定。写好后让我确认。
🧩 和 Claude Code 的对应关系: AGENTS.md 之于 Codex,就等于 CLAUDE.md 之于 Claude Code —— 作用一模一样,只是文件名不同。如果你两个工具都用,可以让其中一份内容互相参考。

💬怎样把话说好

这几条来自 Codex 官方 prompting 文档,是能实打实提高第一次成功率的说话方式。

1

给它一个能自己核对的标准

官方原话:Codex 在能验证自己工作的时候,产出质量更高。所以尽量告诉它:怎么复现问题、怎么算做对了、跑哪个检查能确认。

没有验证标准
把这个函数修一下。
带验证标准
修复 validateEmail 函数。它现在对 'user@.com' 会误判为有效。改完跑 pnpm test 确认所有测试通过,再告诉我结果。
2

把大任务拆成小步

官方原话:任务拆成更小、更聚焦的步骤时,Codex 做得更好 —— 也更容易测试、更容易让你审、更不容易出错。

先做第一小步
我们要把 登录页 从 JavaScript 迁移到 TypeScript。先只处理 这一个文件,改完确认能通过类型检查,我看过之后再继续下一个。
3

不知道怎么拆?让它先出计划

官方建议:如果你自己也不确定该怎么分解,直接让 Codex 先提一个执行计划,你看过、调整过,再让它动手。

先出方案再动手
我想给这个项目 加上深色模式。先别改代码,先看一下现有的样式是怎么组织的,给我一个分步计划。我确认后你再实现。
4

多步骤大目标,用 Goal Mode

对于步骤多、需要明确"完成标准"的活,可以用 /goal 给它一个持久目标。目标要写得具体、可衡量,它会朝这个目标持续推进,你随时能补充约束或暂停。

一个可衡量的目标
目标:把 首页 的可交互时间(TTI)降到 1 秒 以内。你可以自己分析、动手优化,每完成一个改动告诉我效果。
💡 Goal Mode 需要在 config.toml 里打开 features.goals。不确定怎么开?直接问 Codex "帮我开启 goal 功能"。
5

主动喂上下文,别让它猜

提到具体文件、贴上截图、说清楚约束,它的判断质量会明显更好。用 IDE 扩展时,它会自动带上你正打开的文件和选中的代码,省得你复制粘贴。

带足上下文
这是设计稿截图[贴图]。请对照它检查 src/components/Card.tsx 实现出来的效果,列出哪里和设计不一致,再帮我改到一致。
💡 不知道某个功能怎么用?直接问它。 比如"怎么切换模型?""怎么把任务丢到云端跑?",Codex 会直接告诉你。它和 Claude Code 一样,把它当靠谱同事委派,而不是当命令行背指令。
